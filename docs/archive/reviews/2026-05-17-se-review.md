## 1. 项目理解（一段话）

ProbeFlash 经历了从 **v0.3 SPA + SQLite 单体 issue tracker** → **Skill / Bridge / Trail 三 facet** 的产品形态 pivot（D-018）。根因不是工程问题（v0.3 工程完整度极高，已通过 systemd / release tarball 全套部署），而是形态错配 —— 为 5-15 人小作坊做了"跨组需求单"。Pivot 后两条宪法：(1) 输入只能"当下填、当下受益"；(2) **阻塞可见、产能不可比**（D-019）。所有数据 markdown + git native，无 SQLite，无新 server；**飞书作为输入数据层 + 通知层，ProbeFlash 是处理器，不是另一个 UI**。当前：Skill 自用喂养（`debug-checklist` / `personal-daily-summary` / `pre-match-checklist` 三 v0.0.1 已落地），LARK-01~04 pending（plan 已草），Bridge / Trail 等 archive ≥20 条后再启动。

**硬约束**：v0.3 冻结；备赛期不夜跑；冷静期 48-72h；原子任务单 commit；密钥严禁入仓；skill 唯一权威源在 `.agents/skills/<name>/SKILL.md`。

## 4. 设计建议（10 条，按优先级分层）

### 必须改（违反现有 ADR 或会埋雷）

**1. Skill prompt 不要硬编码到 `lark-gateway/src/skills/`**
违反 AGENTS.md §9 "唯一权威源"。改为 `fs.readFileSync('.agents/skills/debug-checklist/SKILL.md')` 取 frontmatter 之后的 body 作 system prompt。需要让 gateway 知道仓库根 — 用 env `PROBEFLASH_ROOT`。

**2. 飞书 API 用官方 SDK `@larksuiteoapi/node-sdk`**
替换手写 ~140 行 `services/feishu-client.ts`。收益：自动 token 刷新、签名验证、card builder（types 完整）、event subscription server。减少 bug 面（重试 / 错误码 / 签名校验）。

**3. webhook idempotency**
飞书失败重试 3 次 / 5s。当前 plan 无 dedup。最小实现：webhook 入口 `if (processedEventIds.has(event.header.event_id)) return ok;`，set 用 `lru-cache`（10k 容量足够），或写 `/tmp/lark-gateway/processed/`。

### 推荐改（更省事 + 更对齐宪法）

**4. Express → Hono（或纯 `node:http`）**
1 个 webhook + 1 个 health endpoint，Hono 更轻 / TS 更好。也可以零依赖直接 `node:http`。

**5. SSE manager 简化**
plan 写了 `createSseStream` / `AsyncIterable<string>` / `connections` map。但飞书卡片更新是 `PATCH /messages/:id` —— 没有 SSE 客户端连接需要管理。砍到只剩 `formatCardMarkdown` + `streamProgressively(messageId, items, updateFn)`。从 ~100 行 → ~30 行。

**6. monorepo + shared package**
v0.3 的 zod schema（D-005）和未来 gateway 的 schema 应共享。`pnpm` workspaces：`packages/shared/types`、`packages/shared/schemas`。短期成本：搭 workspace。长期收益：v0.3 的 `IssueCardSchema` 在 Trail viewer / 任意未来面板都能复用。

**7. workflow protocol 简化或归档**
`docs/agents/workflow/{queue,state}.md` 解决的是"多 agent 并行认领"——当前不存在该场景。建议：
- 短期：标 "experimental, 暂不启用"，lark-gateway 直接用 plans/ + atomic-task 跑
- 长期：真有并行 subagent 需求再补；现在是 over-fit

### 长期方向（备赛后 / Trail 启动时）

**8. Trail = Quartz fork**
archive ≥ 20 条后，clone Quartz 到 `apps/trail-viewer/`，定制：
- 中文界面 / 战队主题
- frontmatter schema 对齐 `.debug-archive/*.md`
- 自定义页：「赛季年鉴」by Claude prompt → markdown → Quartz 渲染

不做：自写静态生成器、用 React 重做、引入向量搜索。

**9. Bridge UI = 飞书多维表格 + ProbeFlash 同步器**
Bridge 启动时**不做新网页**。流程：
- 飞书多维表格 = member/task 真源（队员直接看 / 改）
- ProbeFlash 监听 webhook 同步表格写入 `.bridge/<date>/<task>.md`（markdown audit log）
- "可能能帮上忙"配对建议 → 飞书机器人 @ 相关人
- 完全无"ProbeFlash 网页"角色

**10. 冷静期不动 LARK-01**
lark-gateway plan 已详细到能上手，但备赛期硬约束是 Skill-only。建议：
- 现在不开 `apps/lark-gateway/` 目录（保持 plan 状态）
- 备赛结束后先做 LARK-02 API 调研，根据飞书实际能力修订 plan
- 提前实施会浪费——飞书 API 限制可能让 plan 一半作废
