整体原则是：**先做“项目绑定 + 调试闭环”的最小可用系统；S3 阶段优先把 localStorage 演示版迁移为局域网可访问、服务器长期存储版，不做大而全的知识管理平台。**

------

# 一、产品定位

## 1.1 产品一句话定义

一个**绑定代码仓库的硬件调试闭环局域网工具**：用户在调试现场用极低成本记录碎片问题，系统将“瞬时混乱”收束成“可追踪、可排查、可归档”的问题闭环，并在 S3 阶段优先支持战队同一 WiFi 下共享访问与服务器长期存储。

## 1.2 它不是什么

它**不是普通笔记软件**，也不是泛用待办工具、知识库工具。

S3 当前也**不是** AI / RAG / 权限系统 / Electron / 复杂统计项目；S3 的核心交付是“局域网共享 + 服务器长期存储”。

它的核心不是“记东西”，而是：

- 以**项目仓库路径**为绑定对象
- 以**调试问题卡**为核心实体
- 以**排查推进 -> 结案总结 -> 错误表归档**为主流程
- 以**Git 变更、分支状态、相关文件、历史问题**为上下文增强
- 以**服务器端长期存储**保证战队设备之间可共享、可长期保留
- AI 辅助结构化和收束保留为产品能力方向，但 S3 不优先实现 AI / RAG

## 1.3 产品价值

解决硬件/嵌入式调试中最常见的几个断裂点：

- 现场记录碎片化，后面根本看不懂
- 问题和项目上下文脱钩，回头找不到对应代码与改动
- 排查过程没有沉淀，只记住“修好了”，忘了“为什么坏、怎么修”
- 同类问题反复出现，缺少项目内错误知识库
- Git 改动与调试问题之间没有连接，团队很难复盘

------

# 二、目标用户

## 2.1 第一目标用户

做硬件/嵌入式/机器人/ROS/控制系统调试的人，尤其是：

- 学生竞赛队伍
- 嵌入式开发者
- 机器人调试工程师
- 电子控制/上位机/底盘联调人员

## 2.2 用户特征

这类用户通常有这些特点：

- 手上同时连着板子、串口、示波器、电机、传感器，不方便长篇输入
- 问题出现非常快，记录必须“低中断”
- 问题往往与某个仓库、某个分支、某段改动强相关
- 真正重要的不是“记下来”，而是“最后能形成闭环沉淀”

## 2.3 当前替代方案

用户现在一般用：

- 微信/QQ 发给自己
- VSCode 临时 txt / md
- 纸记
- Notion / Obsidian
- 群里吼一句
- 脑子记

这些方案都无法稳定实现“**项目绑定 + 调试闭环 + repo-aware 分析 + 错误归档**”。

------

# 三、核心场景

## 3.1 场景 A：调试现场快记

用户正在板前调试，出现异常，只来得及输入一句：

- “串口又乱了”
- “自动跑点往后走了”
- “上电后电机初始化偶发失败”
- “CAN 上低 ID 明显被挤掉”

系统立即做几件事：

- 绑定当前仓库
- 自动读取当前分支、最近提交、未提交改动
- AI 将碎片语句补全成问题卡
- 猜测可能方向
- 给出第一轮排查建议

## 3.2 场景 B：持续追记

用户排查过程中不断补记录：

- “换了 TTL 线无效”
- “重启后概率降低”
- “怀疑是串口号漂移”
- “只在 Ubuntu server 上出现”
- “昨天改过底盘坐标转换”

系统把这些记录自动挂到同一问题卡下，并动态更新：

- 当前怀疑方向
- 已尝试动作
- 未验证假设
- 关联文件/关联提交

## 3.3 场景 C：结案沉淀

问题解决后，系统自动生成：

- 错误表记录
- 一篇 markdown 归档文档
- 自动命名
- 可选写入仓库内 docs/debug 目录
- 未来可被检索、复用、关联推荐

## 3.4 场景 D：旧问题复发

用户再次输入“串口又乱了”，系统可根据仓库历史问题提示：

- 之前是否出现过类似问题
- 当时怎么定位
- 最终结论是什么
- 涉及哪些文件/分支/设备环境

------

# 四、MVP 功能范围

这里严格按最小闭环设计。

## 4.1 必做功能

### 1）项目绑定

- 用户首次添加仓库路径
- 系统识别是否为 Git 仓库
- 保存为一个“项目工作区”

### 2）快闪记录入口

- 全局快捷键呼出小窗
- 可快速输入一句碎片描述
- 默认归属当前选中项目

### 3）问题卡生成

- 根据碎片输入创建问题卡
- 自动抓取仓库上下文
- AI 生成结构化问题摘要、初步猜测、排查建议

### 4）问题卡持续追记

- 支持继续追加记录
- 每条记录可标记为：现象 / 怀疑 / 已尝试 / 结论 / 附件补充

### 5）仓库上下文挂载

- 当前分支
- 最近 commits
- working tree 脏状态
- 近期变更文件
- 用户手动勾选的关联文件

### 6）结案总结

- 用户点击“结案”
- AI 自动总结问题闭环
- 生成错误表项
- 生成 markdown 归档文档

### 7）服务器长期存储与归档

- S3 阶段从 `window.localStorage` 迁移到服务端 SQLite / 数据目录
- 按规则持久化 IssueCard / InvestigationRecord / ErrorEntry / ArchiveDocument
- 支持同一 WiFi 下多设备读取同一份服务器数据
- 自动命名，并可从历史列表检索

## 4.2 暂不做

S3 / MVP 先不做这些：

- 多人实时协作
- 云同步
- 自动读取串口日志/示波器数据
- 自动改代码
- 太复杂的图谱系统
- 跨仓库知识推理
- 高级看板和统计大屏
- 真正自治 agent 工作流编排
- S3 阶段内不做 AI / RAG / 权限系统 / Electron / fs/IPC / 大 UI 重构 / 复杂统计

------

# 五、页面结构设计

建议做成 4 个主界面 + 1 个快闪入口。S3 阶段的访问形态优先按浏览器局域网入口设计，而不是桌面 Electron 外壳。

## 5.1 快闪输入窗

用途：极低成本记问题。

内容：

- 项目选择器
- 单行/多行输入框
- 一键“创建问题卡”
- 最近打开的问题卡快捷入口
- 可选标签：硬件 / 串口 / CAN / 底盘 / 传感器 / 上位机

特点：

- 小窗置顶
- 支持全局快捷键
- 默认尽量少字段

## 5.2 项目主页

展示当前项目整体状态。

模块：

- 项目基本信息
- 仓库路径
- 当前分支
- working tree 状态
- 最近问题卡
- 最近归档文档
- 最近关联文件/提交

这是“项目绑定”的核心入口，必须让用户一进来就感到：**这是围绕仓库调试，而不是随便写笔记。**

## 5.3 问题卡详情页

这是核心页面。

建议分 5 个区：

### A. 头部概览

- 问题标题
- 状态
- 严重程度
- 创建时间 / 最后更新时间
- 当前怀疑方向

### B. 现象与摘要

- 原始碎片输入
- AI 整理后的结构化描述
- 影响范围
- 复现条件

### C. 排查时间线

按时间追加记录：

- 现象
- 猜测
- 尝试
- 结果
- 结论

### D. 仓库上下文

- 当前分支
- 最近提交
- 关联文件
- 相关 diff 摘要
- 历史相似问题

### E. AI 辅助区

- 建议排查方向
- 建议读取文件
- 建议补充的信息
- 一键生成阶段总结
- 一键结案

## 5.4 错误表页

以表格/列表方式管理已结案问题。

字段建议：

- 错误编号
- 标题
- 项目
- 类别
- 根因
- 解决方式
- 首次出现时间
- 最近复发时间
- 关联文档

## 5.5 归档文档页

显示 markdown 归档结果，可：

- 预览
- 复制
- 打开文件位置
- 回写到仓库 docs 目录

# 七、AI 功能设计

AI 不应当成为“万能对话框”，而应当成为几个明确的、可控的功能模块。

## 7.1 碎片补全

输入：

- 用户原始碎片文字
- 项目名
- 最近上下文
- 可选历史问题摘要

输出：

- 更完整的问题标题
- 结构化问题摘要
- 现象/影响/复现条件的初稿

例子：

用户输入：
“自动跑点方向不对”

AI 输出：

- 标题：自动跑点阶段底盘运动方向与预期不一致
- 现象：在自动跑点过程中，本应向右移动的车体出现向后运动
- 可能影响：自动路径执行异常，存在撞击风险
- 建议补充：问题是否仅在自动模式出现，手动世界系控制是否正常

## 7.2 问题猜测

目标不是替用户拍板，而是给出**合理怀疑方向列表**。

输出格式建议固定：

- 可能方向 1：坐标系转换错误
- 依据：最近改动涉及 go_path / 底盘坐标转换
- 建议验证：检查自动路径点输入坐标语义与当前雷达安装方向是否一致

要求：

- 每条猜测都要给“依据”和“下一步建议”
- 不输出空泛鸡汤

## 7.3 Repo-aware 分析

输入：

- 当前分支
- 最近提交
- working tree 变更
- 关联文件内容摘要
- 当前问题卡文本

输出：

- 哪些改动可能相关
- 哪些文件应优先查看
- 是否和历史问题类似

这个模块的核心价值是：**让问题与仓库关联起来，而不是只会泛泛猜测。**

## 7.4 结案总结

当用户点击结案时，AI 生成：

- 问题背景
- 现象时间线
- 已尝试方法
- 根因判断
- 最终解决方式
- 后续预防建议

## 7.5 自动命名

要命名的不只是标题，还包括归档文件名。

命名规则建议基于：

- 日期
- 问题类别
- 关键词
- 简短具体

例如：

- `2026-04-20_serial-port-device-drift.md`
- `2026-04-20_auto-path-direction-mismatch.md`
- `2026-04-20_can-bus-low-id-starvation.md`

## 7.6 AI 调用原则

MVP 阶段建议遵循：

- 以模板化 prompt 为主
- 每个 AI 功能单独定义输入输出 schema
- 输出尽量 JSON 化，再渲染到 UI
- 允许用户编辑 AI 输出，不直接当真理

# 八、仓库上下文设计

这是产品区别于普通笔记软件的关键层。

## 8.1 绑定方式

每个项目必须绑定一个本地仓库路径。

首次绑定时做这些校验：

- 是否存在 `.git`
- 是否可读取当前分支
- 是否可执行 Git 命令
- 是否存在建议归档目录

## 8.2 MVP 阶段采集的仓库上下文

优先只做最值钱的几项：

### 基础 Git 信息

- 当前分支
- HEAD commit
- 最近 10 条提交
- working tree 是否有未提交改动
- 最近变更文件列表

### 用户手动选择的关联上下文

- 用户勾选关联文件
- 用户粘贴函数名/模块名
- 用户指定“这次和哪个提交有关”

### 历史问题关联

- 当前项目下的相似问题卡
- 当前项目下的历史错误表项

## 8.3 暂不自动做的事情

MVP 不建议一上来做：

- 全仓 embedding
- 全量代码库语义索引
- 自动解析所有 C/C++ 调用图
- 大规模后台监控

原因很简单：成本高、噪声大、调试价值未必最高。

## 8.4 推荐的 repo-aware 实现顺序

第一版：

- Git CLI + 文件路径 + 历史 markdown 检索

第二版：

- 选中文件摘要
- 基于简单关键词和路径规则推荐关联文件

第三版：

- 轻量代码索引
- embedding 检索
- 提交与问题模式关联分析

------

# 九、状态机设计

产品主状态机围绕“问题卡生命周期”。

## 9.1 问题卡状态

```text
draft -> open -> analyzing -> investigating -> waiting_feedback -> resolved -> archived
```

## 9.2 各状态定义

### draft

用户刚输入碎片，但还没正式生成问题卡。

### open

问题卡已创建，基础上下文已挂载。

### analyzing

AI 正在补全、猜测、生成建议。

### investigating

用户持续追加排查记录中。

### waiting_feedback

问题暂时挂起，等待用户下一次验证、上板、复测。

### resolved

用户确认问题已定位并解决，但还未正式归档。

### archived

已生成错误表项和 markdown 文档，完成闭环。

## 9.3 状态迁移规则

- 快闪输入提交后：`draft -> open`
- AI 分析中：`open -> analyzing`
- 分析完成开始排查：`analyzing -> investigating`
- 用户暂停：`investigating -> waiting_feedback`
- 继续排查：`waiting_feedback -> investigating`
- 用户确认解决：`investigating -> resolved`
- 生成归档：`resolved -> archived`

## 9.4 子状态建议

MVP 不做复杂子状态机，但可以有轻量标记：

- 是否已挂载仓库快照
- 是否已生成 AI 建议
- 是否已生成结案总结
- 是否已写入归档文件

------

# 十、文件归档目录结构和命名规则

必须体现“项目绑定”。

## 10.1 推荐目录

优先推荐归档到仓库内：

```text
<repo>/
  docs/
    debug/
      issues/
      error-table/
```

或者若用户不想污染仓库，可支持外部目录：

```text
<workspace-root>/
  <project-name>/
    issues/
    error-table/
```

## 10.2 问题归档文档结构

每个问题卡生成一个 markdown：

```text
docs/debug/issues/2026-04-20_auto-path-direction-mismatch.md
```

内容模板建议：

```md
# 自动跑点阶段底盘运动方向与预期不一致

## 基本信息
- 项目：
- 仓库路径：
- 分支：
- 创建时间：
- 结案时间：
- 标签：

## 原始问题输入
自动跑点方向不对

## 现象描述
...

## 排查过程
### 1. 初始怀疑
...
### 2. 已尝试动作
...
### 3. 观察结果
...

## 根因
...

## 解决方案
...

## 预防建议
...

## 关联文件
- ...
## 关联提交
- ...
```

## 10.3 错误表文件

错误表可以先维护为一个总 markdown 或 JSON。S3 迁移后，机器可读数据以服务器 SQLite 为主，Markdown / JSON 导出作为可读归档和备份形态。

MVP 更推荐双存储：

- 机器可读：`error-table/errors.json`
- 人类可读：`error-table/README.md`

### errors.json

用于后续检索、相似问题推荐。

### README.md

用于快速浏览。

## 10.4 命名规则

### 问题文档

```
YYYY-MM-DD_<slug>.md
```

例：

- `2026-04-20_serial-device-random-switch.md`
- `2026-04-20_auto-path-direction-mismatch.md`

### 错误编号

```
DBG-YYYYMMDD-XXX
```

例：

- `DBG-20260420-001`

命名原则：

- 日期优先
- 英文 slug 简短具体
- 避免“issue1”“bugtest”这种无信息命名

# 补一个更像产品经理结论的收束

## 这款产品的最小闭环

最小闭环其实只有 6 步：

1. 绑定仓库
2. 快闪输入一句碎片问题
3. 自动生成问题卡
4. 持续追加排查记录
5. AI 结案总结
6. 归档为错误表 + markdown

只要这 6 步打通，这个产品就已经不是“笔记工具”，而是一个真正的**项目绑定式硬件调试闭环系统**。

## MVP 成功标准

如果第一版能做到下面这几件事，就已经有价值：

- 用户愿意在调试现场真的打开它，而不是继续发微信给自己
- 一次问题从出现到归档可以完整走通
- 归档文档回头看得懂，不是 AI 写废话
- 第二次遇到类似问题时，能从历史记录中得到帮助
