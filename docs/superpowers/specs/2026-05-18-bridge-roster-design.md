## 1. 问题陈述与设计目标

> 术语：本文中"出车"指实车占用调试时段（联调期约车上电做测试），不是"出机构"或离开车间。

### 1.1 痛点

1. 一台实车要被 5+ 个机构（机械臂 / 底盘 / 视觉 / 导航 / 自动化测试）调试，车是稀缺资源。
2. 一人调车多人围观，其他人不知道当下该干嘛。
3. 不同组对 git 的使用频率差异极大——只有程序组高频 commit，其他组工作事件（出机构、上电、改硬件、开会决策）与 commit 脱节。
4. 人员临时换手时，接手方找不到"前任卡在哪 / 看哪个仓库 / 找谁要文件"。
5. 联调期"占车时间窗"目前靠口头，事后难追溯。

### 1.2 最强诉求

**直观看到谁负责什么** —— 以及"谁推到哪里、和谁对接、什么时候上车"。

### 1.3 设计目标

- 所有组（不只是程序组）都能填得动、看得懂。
- 填写成本极低（每次推进 ≤ 一句话），并且**填者当下受益**（D-018 宪法 #1）。
- 让"阻塞 / 卡住 / 谁需要帮"可见，但**不做产能排名**（D-019）。
- 不上传任何代码；git 仅作为程序组那一栏的辅助源（只读本地 mirror，可选项，非主线）。

### 1.4 Out of scope

- 飞书多维表格之外的载体（钉钉 / 微信 / 自建 web）。
- 真实 API 调用与同步实现（依赖 LARK-02 完成后单独落地）。
- 产能 / 绩效统计、人与人比较。
- 强制工作流（节点审批、状态机校验、必填校验）。

## 2. 整体形态

```
飞书多维表格（backbone，人填）
  ├─ 表 A: 任务卡         ← "谁负责什么" 的事实源
  ├─ 表 B: 推进记录       ← "谁推到哪里 / 对接 / 接手 / 出车" 的事件流
  └─ 视图（飞书内置过滤器）：
      · 我负责的任务（owner == me）
      · 当前卡住的任务（status == 卡住）
      · 联调期出车窗（progress_log.type == 出车）

ProbeFlash 中央枢纽（轻量 server / cron，备赛后）
  ├─ 拉飞书多维表格 → 本地快照（.bridge/snapshot.yaml + records/*.md）
  ├─ 程序组 git mirror 池（可选，v2 引入）→ 自动写一条 progress_log
  └─ 生成衍生视图：
      · 跨组进度时间线
      · 接手视图（per 任务）
      · 出车时间窗视图（联调期）
```

**同步方向：单向，人填飞书 → ProbeFlash 只读。** 避免双源真相和回写竞争。第 4 步以后再视情况开"程序组 commit → 飞书"的单向写。

## 3. 数据模型（飞书多维表格 schema）

### 3.1 表 A — 任务卡 `tasks`

| 字段 | 类型 | 必填 | 说明 / 例子 |
|---|---|---|---|
| `task_id` | 自增 | 是 | 飞书自动 |
| `task_name` | 文本 | 是 | "机械臂电机调试" |
| `group` | 单选 | 是 | 机械 / 电控 / 视觉 / 导航 / 自动化测试 / 程序 |
| `owner` | 人员 | 是 | 当前主负责人（变更留痕）|
| `liaisons` | 人员（多选） | 否 | 当前正协作的人（"和谁对接"）|
| `status` | 单选 | 是 | 待启动 / 进行中 / 卡住 / 已完成 / 已搁置 |
| `summary` | 文本（短）| 是 | 一句话："PID 在 30Hz 调好，等上电验证" |
| `started_at` | 日期 | 否 | 实际启动日 |
| `expected_done` | 日期 | 否 | 预期完成 |
| `actual_done` | 日期 | 否 | 实际完成（status=已完成 时填）|
| `blocker` | 文本（短）| 否 | status=卡住 时必填："等机械组提供法兰盘 3D 模型" |
| `assets` | URL（多）| 否 | 仓库 / 文档 / 图纸链接 |
| `tags` | 多选 | 否 | 子模块 tag（pid / can-bus / 标定 / ...）|

**飞书内置视图**：
- *我负责的*：`owner == me`
- *卡住的*：`status == 卡住`
- *本周到期*：`expected_done <= now+7d AND status != 已完成`
- *某机构全貌*：`group == 机械臂`

### 3.2 表 B — 推进记录 `progress_log`

| 字段 | 类型 | 必填 | 说明 / 例子 |
|---|---|---|---|
| `log_id` | 自增 | 是 | |
| `task` | 关联 → `tasks` | 是 | |
| `actor` | 人员 | 是 | 这条推进是谁做的（规范层约束：只填自己，飞书 schema 不强制）|
| `at` | 日期时间 | 是 | 默认当前时刻 |
| `type` | 单选 | 是 | **推进** / **对接** / **接手** / **卡住** / **解封** / **出车** |
| `note` | 文本（短）| 是 | 一句话内容 |
| `counterpart` | 人员（多选）| 否 | type=对接/接手 时填（对方是谁）|
| `delivery` | URL | 否 | 交付物链接（commit / 文档 / 图纸）|
| `vehicle_window` | 时间段 | 否 | type=出车 时填："5-20 19:00–21:00 在车间 A" |

**每行语义示例**：
- `推进`: "驱动器 CAN 地址改 0x02，闭环模式跑通"
- `对接`: "和 @李四 约定 视觉→运动 接口字段加 timestamp，5-22 前给"
- `接手`: "@张三 临时去机械组帮忙，我接手；下一步要看 `arm-controller` 仓库 `pid_tuner.cc`"
- `卡住`: "等机械组法兰盘，3 天没动"
- `解封`: "法兰盘到了，恢复推进"
- `出车`: "明晚 19:00–21:00 占车做电机标定，需要电控组配合接线"

### 3.3 模型自洽性说明

- "任务卡里**现在**谁负责什么"→ 一眼可见（呼应"直观看到谁负责什么"）。
- "推进记录按 `at` 倒序"→ **就是**用户描述的"一直推进的进度表"。
- "段"（连续同一 actor 序列）天然形成，飞书按 `task + at` 排序即可得到；不需要单独的"段表"。
- 接手时新人**只读该任务对应的推进记录**就懂前任卡在哪、最近交付了什么、和谁对接过。
- 出车时间窗 = `type=出车` 的 progress_log 行集合，不需要新表。

### 3.4 表 C — 成员表 `members`（可选，备赛后视实际需求决定）

只在飞书多维表格里关联人员字段时才有意义。本 spec 不展开字段细节。

## 4. 填写规范

### 4.1 谁填、何时填、当下回报

| 角色 | 填什么 | 何时 | 当下回报（满足宪法 #1）|
|---|---|---|---|
| 任务负责人 | 创建任务卡（task_name / group / summary / expected_done）| 启动一个新调试事项时 | 自己工作目标显式化；遇阻时能 @ 队长求支援 |
| 任务负责人 | 加一行 progress_log（type=推进，note 一句话）| 取得阶段性进展 / 切换工作 / 每天结束前 | 第二天接着干时不用回忆；周报自动有内容 |
| 任务负责人 | 改任务卡 `status=卡住` + 加 type=卡住 progress_log | 真卡住超过半天 | 队长 / 对接人能看到，主动找人帮 |
| 接手人 | 加 type=接手 progress_log + 改 `owner` | 接手前任工作 | 后续问责清晰；不会重做前任已做的事 |
| 任意人 | 加 type=出车 progress_log | 联调期约车 | 避免撞车，当下回报 = 车被占的概率降低 |

### 4.2 不许的事（防止填了反成包袱）

- ❌ 不要求每天必填——推进了才填，没推进就不填（避免打卡式应付）。
- ❌ 不允许填别人的 progress_log（`actor` 只能是自己；防止代笔扭曲数据；为规范约束，飞书 schema 不强制，靠队内自律 + 飞书内置变更日志兜底）。
- ❌ 不在任务卡里写长篇分析——长内容放 `assets` 链接的文档里。
- ❌ 不做"完成数排名"视图（D-019 红线）。

### 4.3 与现有飞书使用的关系

- 飞书群聊照常使用——多维表格只承载"结构化能复用"的部分。
- 不期望取代群聊"实时通知"功能；状态变更后期可由 ProbeFlash 反推到群里（LARK-04 范畴）。

## 5. 衍生视图（ProbeFlash 生成）

飞书内置视图已能回答"我负责什么 / 谁卡住 / 出车时段"。ProbeFlash 额外生成的衍生视图，承载飞书做不到 / 做得难看的部分。

### 5.1 跨组进度时间线 `views/progress.md`

- 数据源：所有任务卡的 progress_log 按 `at` 倒序聚合。
- 形态：markdown 表格，按"组 → 任务 → 时间"分组。
- 用途：周回顾时一眼看到"这周整队推到哪里了"。
- 渲染时机：每天凌晨 cron / 手动 `bridge sync`。

### 5.2 接手视图 `views/handover/<task-id>.md`

- 数据源：单个任务卡 + 它全部的 progress_log。
- 形态：
  - 顶部：任务基本信息 + 当前负责人 + 关联资产链接
  - 中间：负责人变更序列（从 type=接手 行推出）
  - 底部：最近 N 条推进记录（默认 10）+ 卡住原因
- 用途：接手时**只看一个文件**就懂。
- 触发：人工 / 接手 type 写入后自动生成。

### 5.3 出车时间窗视图 `views/vehicle-window.md`

- 数据源：`progress_log where type == 出车`。
- 形态：日历视图（按时间排序），冲突高亮（时段重叠标红）。
- 用途：联调期一眼避撞车；冲突时由队长协调。
- 状态：在 forward-looking 阶段不强求实现冲突算法，列表呈现即可。

### 5.4 不做的视图（红线）

- 不做"人员完成数 / 推进数 / 任务数"排行（D-019）。
- 不做"个人效率分"。
- 不做基于 progress_log 条数的活跃度可视化。

## 7. 风险与开放问题

### 7.1 主要风险

| 风险 | 说明 | 缓解 |
|---|---|---|
| **非程序组不填多维表格** | 这条不破，整个 backbone 就空 | 第 0 步先贴墙纸面跑通；不填则本 spec 直接退回不再投入 |
| **飞书 API 权限不足** | LARK-02 可能发现读不到多维表格 / 写不动群消息 | 本 spec 顶部明确依赖 LARK-02；不行就退到纯飞书内 + 手动导出 |
| **填写变成应付** | "每天必填"会催生水推进 | §4.2 已明确：推进了才填；不做活跃度排行 |
| **多维表格性能** | 大量 progress_log 后变慢 | 单赛季数据规模 < 几千行，飞书表格能扛；不行就分表（按月 / 按组）|
| **责任段争议** | progress_log 的 actor 可篡改飞书行 | 飞书内置变更日志；本 spec 不引入额外审计 |

### 7.2 开放问题（spec 落地前需要拍板）

1. **任务卡的"已搁置" vs "已完成" 边界**：搁置一阵又重启，算同一卡还是新建？倾向"同一卡，加 type=解封 推进记录"。
2. **进度摘要 `summary` 是否随推进自动更新**：手动改 vs 取最新一条 progress_log 的 note？倾向手动（避免覆盖人脑提炼）。
3. **进度表跨赛季归档策略**：赛季结束后是否冻结一份只读快照（喂给 Trail）？延后到 Trail 启动时再拍。
4. **出车冲突检测**：是否需要在 §5.3 加入"约车前自动检查时段冲突"的逻辑？倾向第一版不做，由人脑负责。
5. **隐私 / 敏感字段**：飞书表格全员可见，是否有不该曝光的字段？倾向无；任务都是工程性的。
